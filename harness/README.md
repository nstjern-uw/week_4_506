# Race condition harness

Empirical proof that the save/publish bug in `app/server.js` is a race
condition between an in-flight `POST /draft` and a `POST /publish`.

## Files

- `run-race.js` ﻗ°½ boots `app/server.js` in a child process with `TRACE=1`,
  drives the "save A ﻗ│φ save B (no await) ﻗ│φ publish" scenario, captures
  the server's NDJSON trace from stdout, and reports per-run results.
  Writes the captured trace to `harness/trace.ndjson`.
- `analyze-trace.js` ﻗ°½ reads `trace.ndjson` and distinguishes unsafe
  races from protected interleavings. A protected interleaving is when
  `publish.received` arrives while a draft is in flight, but publish
  waits until that draft commits before completing.
- `trace.ndjson` ﻗ°½ generated by `run-race.js`. Not committed by hand.

## Quick start

```bash
npm install                        # if you haven't already
node harness/run-race.js           # 5 runs, default timing
node harness/analyze-trace.js      # reads harness/trace.ndjson
```

Expected against the broken code:

- `run-race.js` prints a summary table where `stale: true` for at least
  some rows and a final line `race reproduced: N/5 runs`.
- `analyze-trace.js` prints the event timeline, then a verdict block
  with `Unsafe races: N` and exits non-zero.

Expected after the fix:

- `run-race.js` prints `stale: false` for every row and
  `race reproduced: 0/N runs`.
- `analyze-trace.js` may still show that publish arrived while a draft
  was in flight, but those rows are reported as protected interleavings
  because publish waited for the pending commit before completing.

## Knobs

| Flag | Default | What it does |
|------|---------|--------------|
| `--runs N` | 5 | Number of save+publish trials per invocation |
| `--gap-ms N` | 0 | Wait between sending save B and sending publish |
| `--save-delay-ms N` | 300 | Server-side `SAVE_COMMIT_DELAY_MS` for this run |
| `--port N` | 4100 | Port for the harness server (separate from the dev server) |

To explore the threshold at which the race stops appearing, sweep `--gap-ms`:

```bash
node harness/run-race.js --runs 10 --gap-ms 0
node harness/run-race.js --runs 10 --gap-ms 100
node harness/run-race.js --runs 10 --gap-ms 250
node harness/run-race.js --runs 10 --gap-ms 400   # > save-delay-ms: race should disappear
```

## How the instrumentation works

`app/server.js` emits one JSON object per line to stdout when started
with `TRACE=1`:

| event | fields | when |
|-------|--------|------|
| `server.listening` | `port`, `saveCommitDelayMs` | once at startup |
| `reset` | ﻗ°½ | on `POST /reset` |
| `draft.received` | `reqId`, `content`, `currentDraft` | when `/draft` enters its handler |
| `draft.committed` | `reqId`, `content`, `before`, `after` | when the setTimeout fires |
| `publish.received` | `reqId`, `currentDraftSnapshot` | when `/publish` enters its handler |
| `publish.waitedForDrafts` | `reqId`, `currentDraftSnapshot` | after `/publish` waits for pending saves |
| `publish.completed` | `reqId`, `published` | after `/publish` writes |

`reqId` is a UUID per request so a draft's `received` and `committed`
events can be paired unambiguously even when many are interleaved.

Tracing is gated on `TRACE=1` so `npm test` and normal `npm start` are
unaffected ﻗ°½ the regression test in `tests/race.test.js` and CI run
without any extra output.

## Why both a runner and an analyzer

The runner answers "did publish return stale data?" ﻗ°½ useful, but not
proof of a race on its own. The analyzer answers "did publish complete
unsafely while a draft was provably in flight, or did it wait?" Having both means the
harness can:

1. Quantify how often the bug occurs at a given timing.
2. Show, from the server's own log, the exact event ordering that
   caused each stale publish or confirmed the fix.

## Reproducing what's in the bug report

The user's repro from the README is:

> Type `hello`, click Save, type ` world`, click Publish within ~200ms.

The harness automates that exact pattern: it commits `draft A`, fires
a non-awaited save for `draft B`, and immediately fires publish. Against
the broken code, the default `--save-delay-ms 300 --gap-ms 0` reproduces
the race. After the fix, the same scenario should publish `draft B`.
